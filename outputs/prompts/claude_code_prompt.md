# 철골 부호 카운팅 시스템 — 점진적 일반화 구현 요청

## 프로젝트 배경

건설 도면(DXF)에서 철골 부재 부호의 개수를 자동 카운팅하는 시스템을 만드는 중이야. 최종 목표는 "철골 총 중량(kg) 산출"이지만, 그 첫 단계로 **부호별 개수 카운팅**을 먼저 정확하게 맞추려고 해.

지금까지 진행한 것:
- `app.py`: Streamlit UI, 사용자가 박스 그리면 그 영역의 부호 카운트
- `counter.py`: ezdxf로 TEXT/MTEXT/INSERT 엔티티에서 부호 추출
- `coord_utils.py`, `extent.py`: 좌표 변환
- 정답지 `도면_정답지.xlsx`: 도면 1, 2, 4에 대해 사람이 손으로 만든 정답
- 도면 파일: `도면1.dxf`, `도면2.dxf`, `도면4.dxf` (도면 3, 5는 이후 추가)

## 이번 요청의 핵심

**점진적 일반화(progressive generalization) 방식**으로 시스템을 발전시키고 싶어. 도면 1부터 시작해서 정확도를 맞춘 다음, 도면 2, 4를 하나씩 추가하면서 일반화하는 거야. 이때 반드시 지켜야 할 함정 회피 원칙이 있어.

### 함정 1 — Catastrophic Regression 방지

도면 1만 보고 짠 규칙을 도면 2에 맞추다 보면 도면 1 정확도가 떨어지는 현상. 이걸 막으려면 **회귀 테스트가 자동화**돼야 해. 새 코드 짤 때마다 도면 1~N 모두 통과하는지 즉시 알 수 있어야 함.

### 함정 2 — 도면 1만의 우연을 일반 규칙으로 오인

"도면 1에서 부호가 TXT 레이어에 있다"는 사실은 도면 1만의 특성이지 일반 규칙이 아냐. 도면 2는 레이어 이름이 다를 수 있어. 이런 도면 특성은 **코드가 아니라 데이터(설정 파일)**에 두어야 해.

## 정답지 구조

`도면_정답지.xlsx`는 세 시트(`도면1`, `도면2`, `도면4`)로 구성. 각 시트는 다음 매트릭스 형태야:

```
              | MC1 | MC2 | SC1 | SG1 | ... | 합계
도면명1       |  12 |  6  |     |     |     |  18
도면명2       |     |  4  |     |     |     |   4
...
합계          |     |     |     |     |     |
```

- **행 = 도면 페이지** (예: "(1동)기둥주심도-1", "(2동)지붕구조평면도")
- **열 = 부호** (MC1, MC2, SC1, SG1, ... BR2)
- **셀 = 해당 페이지에서 해당 부호의 개수** (빈 셀 = 0개)

도면 1은 17페이지 × 22부호 (가장 복잡), 도면 2/4는 2페이지 × 9부호 (단순).

### 정답지가 주는 중요한 정보

1. **도면 페이지 단위로 분리**돼있음. "기둥주심도"와 "기둥부호도"는 같은 기둥을 다른 시점에서 보여주는 도면이라 **둘 다 카운트하면 더블카운팅됨**. 정답은 둘 다 12로 적혀있는데, 견적할 때는 한쪽만 사용. 이 점을 시스템이 어떻게 처리할지 설계해야 함.
2. 도면 종류별로 등장 부호가 다름:
   - 기둥주심도/부호도 → MC, SC, BR
   - 바닥보복도 → SG, SB, RSG, RSB
   - 골구도 → VG, VT, WG, BR
   - 기초구조도 → 카운트 제외 (모두 빈 셀)

## 구현 요청

### Step 1 — 테스트 인프라 구축 (가장 먼저)

`tests/` 디렉토리를 만들고 다음을 구현해줘:

**1-A. 정답지 로더** (`tests/ground_truth.py`)
- `도면_정답지.xlsx`를 파싱해서 다음 형태의 dict 반환:
  ```python
  {
      "도면1": {
          "(1동)기둥주심도-1(1~3층 기둥주심도)": {"MC1": 12, "MC2": 6},
          "(1동)기둥주심도-2(옥상층 기둥주심도)": {"MC2": 4},
          ...
      },
      "도면2": { ... },
      "도면4": { ... },
  }
  ```
- 빈 셀(None)은 제외, 합계 행/열도 제외
- 페이지명에서 동(棟) 정보, 도면 종류(기둥주심도/바닥보복도/...) 등을 메타데이터로 함께 추출하면 좋음

**1-B. 회귀 테스트 러너** (`tests/test_regression.py`)
- pytest 기반
- 각 도면 × 각 페이지에 대해 시스템 카운트와 정답을 비교
- 오차율 5% 이하면 통과 (단, 개수가 5 이하인 경우는 ±1까지 허용 — 작은 수에서 5%는 너무 빡빡함)
- 실패 시 상세 리포트: 어느 도면의 어느 페이지에서 어느 부호가 몇 개 차이 나는지
- `pytest -v` 로 실행하면 도면별/페이지별/부호별 통과 여부가 한눈에 보이도록

**1-C. 베이스라인 측정 스크립트** (`tests/baseline.py`)
- 현재 `counter.py` 그대로 사용해서 도면 1 전체 페이지에 대한 카운트 실행
- 정답과 비교한 표 출력: `[페이지명] [부호] [예측] [정답] [차이] [상태]`
- 전체 정확도 (정답과 일치하는 셀 비율), 평균 오차율 계산
- **이 단계는 코드 수정 없이 그냥 현재 시스템이 얼마나 맞히는지 측정만 함**

### Step 2 — 데이터/코드 분리

도면별 특성을 코드에 박지 말고 설정 파일로 빼:

**2-A. 도면 메타데이터** (`config/drawings.yaml` 또는 `.json`)
```yaml
도면1:
  file: 도면1.dxf
  pages:
    - name: "(1동)기둥주심도-1(1~3층 기둥주심도)"
      type: 기둥주심도
      bbox: null  # 처음엔 비워두고 나중에 자동 감지/수동 입력
      expected_symbols: [MC, SC, BR]
    - name: "(1동)2층바닥보복도"
      type: 바닥보복도
      expected_symbols: [SG, SB]
    ...
도면2:
  ...
```

**2-B. 도면 종류별 카운팅 대상 부호 prefix** (`config/symbol_rules.yaml`)
- 기둥주심도/부호도 → MC, SC, BR
- 바닥보복도 → SG, SB
- 옥상바닥보복도 → RSG, RSB
- 골구도 → VG, VT, WG, BR
- 구조평면도 → 페이지마다 다름 (열린 카테고리)
- 기초구조도 → 빈 카테고리

**2-C. 더블카운팅 정책** (`config/dedup_policy.yaml`)
- 같은 부재가 두 도면에 등장하는 케이스 정의
- 예: "기둥주심도와 기둥부호도는 같은 기둥 → 한쪽만 사용"
- 정답지 보면 둘 다 12로 적혀있는데, 시스템 출력은 어느 쪽을 쓸지 명시

### Step 3 — 도면 1 베이스라인 개선

베이스라인 측정 결과를 보고 가장 큰 실패 원인부터 하나씩 잡아.

**중요한 원칙**:
- 코드를 고치기 전에 항상 회귀 테스트를 먼저 돌려서 현재 상태를 기록할 것
- 한 번에 하나의 가설만 검증 (여러 개 동시에 고치면 뭐가 효과 있었는지 모름)
- 도면 1만의 특성으로 보이는 건 코드가 아니라 메타데이터/설정으로 처리

예상되는 실패 케이스:
- 도면 페이지 영역 자동 감지가 안 됨 → 일단 메타데이터에 bbox를 수동 입력하는 방식으로 회피
- 일람표 영역의 부호가 카운트됨 → 페이지별로 일람표 bbox 제외 영역 지정
- 단면도/상세도의 부재 규격 표기가 카운트됨 → `H-200x100x5.5x8` 같은 규격 패턴 제외 정규식

### Step 4 — 점진적 일반화 (도면 2, 4 순차 추가)

도면 1이 정답지 기준 5% 이하 오차 달성하면 도면 2 추가. 이때 반드시:

1. 도면 2 코드/설정 추가 *전*에 회귀 테스트 실행 → 도면 1 통과 확인
2. 도면 2 카운트해보고 실패 케이스 분석
3. 실패 원인 분류:
   - **변형(데이터 문제)**: 레이어 이름 다름, 부호 표기 변형 → 설정 파일에 추가
   - **새 규칙 필요**: 1대1 매칭이 아닌 새 카운팅 패턴 → 새 함수 작성
   - **도면 자체 이상**: 텍스트 explode됨 → 별도 처리 경로 (지금은 일단 skip하고 기록)
4. 변경 후 회귀 테스트 실행 → **도면 1·2 모두 통과해야 함**
5. 도면 1만 깨졌다면 변경을 되돌리고 다른 접근 시도

## 출력 요청

다음 순서로 작업해줘:

1. **현재 코드 분석**: `app.py`, `counter.py`, `coord_utils.py`, `extent.py`, `test_extent.py`를 읽고 현재 구조 요약
2. **Step 1 구현**: `tests/ground_truth.py`, `tests/test_regression.py`, `tests/baseline.py` 작성
3. **베이스라인 측정 실행**: `python tests/baseline.py 도면1` 돌려서 현재 시스템이 도면 1을 얼마나 맞히는지 결과 보고
4. **분석 리포트**: 베이스라인 결과를 바탕으로
   - 어떤 페이지/부호가 잘 맞고, 어떤 게 안 맞는지
   - 안 맞는 케이스의 추정 원인 (페이지 영역 분리 안 됨, 일람표 카운트됨, 등)
   - Step 3에서 가장 먼저 고쳐야 할 항목 우선순위
5. **Step 2 설정 파일 초안**: `config/drawings.yaml`, `config/symbol_rules.yaml` 초안 작성 (도면 1 부분만)

Step 3, 4는 이 첫 라운드 결과를 보고 다음 대화에서 진행.

## 작업 시 주의사항

- **기존 `app.py`, `counter.py`는 건드리지 마**. 새 모듈로 작업하고, 필요하면 `counter.py`를 import해서 사용. 나중에 회귀 테스트로 검증된 시점에만 기존 파일 수정.
- 테스트는 **결정론적**이어야 함. LLM 호출 같은 비결정 요소 넣지 말 것. 순수 ezdxf + 룰 기반.
- 페이지 영역 자동 감지(DBSCAN 등)는 어려운 문제니까 Step 1에서는 시도하지 말고, 메타데이터에 수동 bbox를 적는 방식으로 우회. 자동 감지는 별도 단계.
- 정답지 셀 값이 `None`인 경우와 `0`인 경우를 구분:
  - `None`: 그 페이지에 그 부호 자체가 없음 (카운트 안 함)
  - `0`: 그 페이지에 그 부호가 등장하긴 하는데 0개 (예: 도면4의 지붕층 SC1: 0 — 기둥은 1층에만 있음)
  - 도면 4에 0이 명시된 케이스가 있으니 확인

## 참고 — 도메인 지식

- **철골 부호 화이트리스트**: MC, SC, SG, SB, RSG, RSB, VG, VT, WG, CRG, MT, BR (전부 영문 + 숫자 형태)
- **제외 대상**: F(기초), HD/D/SR/SBR(철근), DS(상세 참조), THK(두께 표기), MF(매트기초), H-숫자(규격 표기)
- **재료 표시 SS275**: 한국 KS 강재 표시. 이 텍스트가 있는 영역은 일람표일 확률 높음
- **일람표 vs 평면도**: 일람표는 격자 모양 + 헤더(부호, 단면치수, MAT'L) 있음. 평면도는 부호가 흩어져 분포
- **단면도/상세도**: `H-200x100x5.5x8` 같은 규격 텍스트가 화살표 끝에 있음. 카운트 대상 아님 (사양 설명)

이대로 작업 시작해줘.
