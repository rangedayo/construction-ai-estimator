# 건설파트너

## **건설파트너의 핵심 니즈**

견적서 한 장 제작에 엄청난 비효율과 피로도가 있음. 견적서를 만들기 위해 방대한 양의 자료를 사람이 일일히 분석해왔다. 이러한 힘듦을 덜기 위한 ai 에이전트를 만들고 싶다.

- ***현직 건설 엔지니어들이 공사비를 뽑기 위해 수행하는 수작업 과정***
    
    ***1단계: 설계 도면 및 스펙 서류 분석 (해독 단계)***
    
    먼저 수백 장의 도면과 수천 장의 서류를 펼쳐놓고 프로젝트의 전체 규모를 파악한다.
    
    - 285장의 CAD 파일을 열어 평면도, 단면도, 구조도를 대조하며 건물의 형태를 머릿속에 입체적으로 그린다.
    - 2,587장의 발주 Spec 서류를 읽으며 어떤 등급의 철근을 쓸지, 어떤 브랜드의 마감재를 쓸지 상세 기준을 확인한다. 이 기준에 따라 단가가 달라지기 때문에 매우 꼼꼼히 읽어야 한다. 
    → 엔지니어가 자재를 새로 선택하는 것이 아니라, 설계자가 지정해둔 상세 사양(Specification)을 정확히 읽어내어 견적서에 반영하는 과정
    
    ***2단계: 레이어 정리 및 벽체 직접 그리기 (트레이싱 단계)***
    
    CAD 파일은 정보가 너무 많아 복잡하므로, 필요한 데이터만 추출하기 위한 밑작업을 한다.
    
    - 레이어 On/Off : 불필요한 가구나 글자 레이어는 끄고, 벽체나 기둥 레이어만 남긴다.
    - 직접 선 따기(Tracing) : 도면에 그려진 선들이 데이터로서 완벽하지 않은 경우가 많다. 엔지니어가 직접 마우스로 벽면을 따라 선을 다시 그리거나(폴리라인 생성), 기둥의 범위를 지정해 면적을 계산할 수 있는 '닫힌 도형'으로 만든다.
    
    ***3단계: 공종별 물량 산출 (카운팅 및 계산 단계)***
    
    이제 본격적으로 '개수'와 '양'을 세기 시작한다. 이 과정이 가장 고통스러운 구간임.
    
    - 기둥 및 창문 개수 세기 : "이 층에 기둥이 몇 개인가?", "창문은 규격별로 몇 개씩인가?"를 하나씩 클릭하며 센다.
    - 길이 및 면적 계산 : 벽의 총 길이, 바닥의 총 면적을 구해 콘크리트 타설량이나 벽지 소요량을 계산한다.
    - 단위 변환 : 도면상의 수치를 실제 공사 현장에서 사용하는 단위(예: 철근의 kg/ton, 콘크리트의 $m^3$)로 변환하여 엑셀에 기록한다.
    
    ***4단계: 엑셀 내역서 작성 및 단가 매핑 (정리 단계)***
    
    산출된 물량을 건설사 표준 양식인 엑셀 파일로 옮긴다.
    
    - 자재 그룹핑 : 동일한 자재지만 설계사마다 다르게 표현한 용어들을 표준 용어로 통일하여 그룹핑한다.
    - 과거 데이터 매핑 : 이전 프로젝트에서 사용했던 유사한 자재의 단가를 찾아와 현재 물량에 곱한다.
    
    ***5단계: 누락 재검증 (검수 단계)***
    
    사람이 하는 일이라 실수가 반드시 나오기 때문에, 처음부터 다시 검토한다.
    
    - 누락 확인 : "기둥 하나를 빼먹지 않았나?", "수치 단위를 착각하지 않았나?"를 확인한다.
    - 리스크 체크 : 물량이 조금이라도 누락되면 수억 원의 손해가 발생하므로, 여러 명의 엔지니어가 서로의 결과물을 교차 검증한다.
- ***두 가지 구체적 프로젝트 과제***
    
    **문제정의 1 : 건축 도면 기반 물량 산출**
    
    실제 건설사의 비공개 CAD 도면에서 콘크리트/철근 물량, 아파트 세대 내 거실·방·창문 개수와 면적, 지하주차장 기둥 수, 계단 수 등을 자동으로 산출하는 AI를 만드는 것이다.
    
    핵심 평가 기준은 단일 도면에서의 높은 정확도보다 다양한 설계사의 다양한 도면 패턴에 대해 안정적이고 편차가 적은 정확도를 내는 것이며, 목표 오차율은 5% 이하.
    
    "방 3개짜리 아파트 세대의 거실 면적과 창문 개수를 AI가 즉시 산출"해준다면, 담당자는 단순 계산에 진을 빼는 대신 "이 공사 비용으로 수주를 진행할 것인가?" 또는 "수익성을 어떻게 높일 것인가?"와 같은 전략적인 의사결정에 더 많은 시간을 쓸 수 있게 된다.
    
    ☞ 만약 이 문제를 해결한다면?
    
    1. 엔지니어 10명이 285장의 CAD 도면을 열어 기둥 하나하나를 클릭하고 엑셀에 수치를 옮겨 적으며 10일 동안 작업하던 걸 
    → AI가 전체 도면을 스캔하여 기둥의 규격(H-Beam 등)과 위치를 초 단위로 파악하고 엑셀 파일을 자동 생성한다. 사람은 AI가 뽑아낸 리스트에 오류가 없는지만 최종 확인
    2. A 설계사는 기둥을 파란색 실선으로, B 설계사는 보라색 점선으로 그리는 등 패턴이 제각각이라 새로운 도면을 볼 때마다 사람이 다시 적응 
    → 프로젝트의 목표인 '안정적인 정확도'가 확보되면, AI가 어떤 설계사의 도면이 들어와도 유연하게 인식해서 도면 패턴이 달라질 때마다 정확도가 널뛰지 않음.
    3. AI가 5% 미만의 오차율로 물량을 산출하면, 최소한 대형 사고(수량 누락 등)는 확실히 잡아낼 수 있다.
    
    **문제정의 2 : 공사 견적서 기반 단가 매핑**
    
    견적서 엑셀 데이터에서 
    
    1. 같은 의미인데 다른 단어로 표현된 자재 분류, 그룹핑
    2. 비슷한 공사 현장 타입에서 함께 사용되는 자재 그룹핑
    3. 신규 견적 요청 시 과거 유사 데이터 매핑, 추천을 수행한다.

## 건축 도메인 필수 개념

### CAD 파일 구조

1. 파일 포맷
    - DWG (DraWinG) : 오토캐드(AutoCAD)의 표준 형식으로, 데이터를 압축해서 저장하기 때문에 용량이 작고 처리 속도가 빠르다.
    - DXF (Drawing eXchange Format) : 서로 다른 CAD 프로그램 간에 데이터를 주고받기 위해 만든 '공용 언어'다. 텍스트 형태(ASCII)로 저장되어 있어 AI가 정보를 읽어오기에 더 수월할 때가 많다.
2. 레이어(Layer) :
    
    도면 위에 겹쳐진 '투명한 층'이다. 예를 들어 '벽체 레이어', '창문 레이어', '전기 배선 레이어'를 각각 따로 만든다. 물량 산출 시 '벽체 레이어'만 켜서 분석하면 훨씬 빠르고 정확하게 데이터를 추출할 수 있다.
    
3. 엔티티(Entity)와 폴리라인(Polyline) :
    - 엔티티 (Entity) : 도면을 구성하는 가장 기본 단위인 점, 선, 원 등을 말한다.
    - 폴리라인 (Polyline) : 여러 개의 선(Line)과 곡선(Arc)이 하나로 연결된 덩어리다.
    
    방의 면적을 구할 때 일반 선 4개로 그리는 것보다, 하나로 연결된 '폴리라인'으로 그리면 클릭 한 번에 면적 데이터가 바로 계산된다. 이번 프로젝트에서 면적 산출의 핵심이 된다.
    
4. 블록 (Block) : 도장(Stamp)
    
    변기, 세면대, 문, 기둥처럼 자주 쓰이는 형태를 하나로 묶어둔 '부품'을 말한다. 도면 안에 창문이 100개 있다면, 일일이 그리는 대신 '창문 블록'을 100번 찍어냄.
    
    프로젝트 제안서에서 언급된 '창문 개수 산출'이나 '기둥 수 산출'은 사실 AI가 이 '블록' 데이터의 이름과 개수를 찾아내는 과정이라고 볼 수 있다.
    

### 건축 도면 종류

![image.png](image.png)

1. 평면도 (Plan) : 
    
    건물을 일정한 높이(보통 바닥에서 1.2m 정도)에서 가로로 잘라 하늘에서 내려다본 그림이다. 방의 배치, 창문과 문의 위치, 벽체의 두께 등을 한눈에 볼 수 있어 면적 산출의 핵심이 됨.
    
2. 입면도 (Elevation) : 
    
    건물을 밖에서 전후좌우 겉모습을 바라본 그림. 건물의 전체적인 모양, 외벽 마감재, 창문의 높이 등 외관 정보를 담고 있음.
    
3. 단면도 (Section) : 
    
    건물을 수직으로 싹둑 잘라 옆에서 본 그림. 층고(층 높이), 슬래브(바닥판)의 두께, 천장 내부 구조 등 수직적인 공간 정보를 확인하기 위해 필수적이다.
    
4. 구조도 : 
    
    건물의 뼈대를 보여주는 도면. 제안서에 나온 H-Beam, 철근, 콘크리트의 규격과 배치가 상세히 적혀 있어, 하중을 견디는 핵심 자재의 물량을 산출할 때 가장 중요하게 쓰인다.
    
5. 설비도 : 
    
    물이 흐르는 배관(급수, 배수)이나 냉난방 덕트 등의 경로를 그린 도면. 제안서 예시 중 '설비 공종'의 견적을 낼 때 이 도면을 분석한다.  
    
6. 전기도 : 
    
    조명, 콘센트, 스위치의 위치와 전선이 지나가는 길을 표시한 도면이다. 건물에 들어가는 전선량이나 전기 장비의 개수를 파악하는 데 사용됨.

### 물량 산출(Quantity Take-Off, QTO)

도면에서 자재의 양을 계산하는 작업. 
콘크리트는 부피(m³), 철근은 무게(kg/ton), 벽체는 면적(m²)으로 산출

### 공종(공사의 종류) 구분

하나의 건물을 짓기 위해 수행되는 수많은 세부 작업들을 전문 분야별로 나눈 카테고리를 말하며, 이를 구분할 줄 알아야 한다.

- 정확한 견적 산출 : 각 공종마다 들어가는 자재(철근 vs 전선)와 전문 인력이 완전히 다르기 때문에, 이를 세분화해야만 정확한 공사비를 계산할 수 있다.
- 노동 집약적 분석 : 현재는 엔지니어들이 285장의 도면을 보며 이 25개 공종별로 들어가는 물량을 일일이 따로 뽑아내야 한다. 제안서에서 "100 Man-Days"가 소요된다고 한 이유도 바로 이 복잡한 분류 작업 때문.

### 구조선과 장식선 처리

1. ***구조선 (Structure Line) → 모델의 '입력(X)'***
    
    구조선은 건물의 뼈대(벽, 기둥, 보 등)를 결정하는 결정적인 정보다. AI가 이 선들을 보고 건물의 기하학적 형태를 해석(Parsing)해야 한다.
    
    구조선을 모델의 메인 입력값($x$)으로 사용하여, AI가 이 선들 사이의 관계(GNN 등)를 통해 공간의 면적과 물량을 계산하도록 유도한다.
    
2. ***장식선 (Decorative/Annotation Line) → 메타데이터(JSON Label)로 따로 저장***
    
    어노테이션(치수선, 지시선, 해치 등)은 형태를 결정하진 않지만, 그 형태에 '의미'를 부여하는 보조 정보다. "이 선은 300mm 두께의 콘크리트 벽이다"라는 정보를 주는 텍스트나 기호들을 말한다.
    
    이를 입력 이미지($x$)에 그대로 두면 모델이 노이즈로 인식할 수 있기 때문에, 이 정보들을 따로 추출해 JSON 형태의 레이블(Label)로 관리해보는 거다.
    

**이렇게 처리했을 때의 장점**

- 복잡한 도면에서 AI가 '치수선'을 '벽'으로 착각하는 실수를 원천 차단한다.
- AI는 뼈대(X)만 보고 빠르게 판단하고, 상세 스펙(JSON)은 나중에 매칭만 하면 되므로 연산 효율이 엄청나게 좋아진다.
- 나중에 수치가 틀렸을 때, "구조선 인식이 틀렸는지" 아니면 "JSON 매핑이 잘못됐는지"를 명확히 디버깅할 수 있다.

## 건축 관련 CV 필수 개념 (문제정의 1 관련)

### 이미지 전처리

CAD 파일(DWG, DXF)을 이미지로 변환했을 때, 불필요한 정보는 버리고 특징만 살리는 작업

- 이진화(Binarization) : 도면에는 다양한 색상이나 회색조가 섞여 있을 수 있어, 이를 오직 흑(0)과 백(255)으로만 나누는 작업이다. 배경은 하얗게, 벽체나 기둥 같은 선은 검게 만들어 AI가 '형태'에만 집중하게 한다.
- 노이즈 제거(Noise Removal) : 도면을 스캔하거나 변환할 때 생기는 자잘한 점들이나 먼지 같은 '잡티'를 지우는 과정. 이게 없어야 AI가 작은 점을 기둥으로 착각하지 않는다.
- 모폴로지 연산(Morphology) : 이미지의 형태(Morph)를 변형하여 노이즈를 없애거나 특징을 강화하는 작업이다. 도면을 스캔하거나 이미지로 변환하면 선이 미세하게 끊기거나 글자가 뭉치는 경우가 많은데, 이를 보정해 준다.
    - 팽창(Dilation) : 선을 한 겹 두껍게 만든다.
        - 용도 : 도면에서 아주 얇게 그려진 벽체 선이나 미세하게 끊어진 H-Beam의 외곽선을 하나로 이어줄 때 사용한다. 그래야 AI가 이를 '닫힌 면적'으로 인식할 수 있음.
    - 침식(Erosion) : 선을 한 겹 깎아낸다.
        - 용도 : 선들이 너무 다닥다닥 붙어서 하나의 덩어리처럼 보일 때, 살을 깎아서 각각의 선(예: 창문 틀의 세부 선들)을 분리할 때 사용한다.
    - 열기(Opening) & 닫기(Closing)
        - 열기(침식 후 팽창) : 작은 점(노이즈)을 먼저 지우고 다시 원래 크기로 복구한다. 도면 위의 먼지나 불필요한 기호를 지울 때 좋다.
        - 닫기(팽창 후 침식) : 끊어진 틈을 먼저 메우고 다시 원래 크기로 줄인다. 벽체 선 사이의 미세한 구멍을 메울 때 필수적임.

### 객체 탐지(Object Detection)

AI가 도면 속에서 "이건 기둥이고, 저건 창문이야"라고 이름표를 붙여주는 단계

1. ***YOLO 계열 (You Only Look Once)*** 
    
    이름처럼 도면 전체를 '딱 한 번만 슥 보고' 객체를 찾아내는 방식이다. 속도가 굉장히 빨라서 수백 장의 도면을 초 단위로 스캔할 때 유리하다. 기둥이나 문 같은 반복되는 '블록'을 빠르게 찾는 데 적합함.
    
2. ***Faster R-CNN*** 
    
    도면을 꼼꼼하게 훑어보며 객체가 있을 법한 위치를 먼저 제안받고 확인하는 방식. YOLO보다 조금 느릴 수 있지만, 아주 작은 기둥 규격이나 정밀한 심볼을 찾을 때 정확도가 높다. 오차율 5% 미만을 달성하기 위해 정밀 검수가 필요한 부분에 쓰일 수 있음.
    
3. ***DETR (DEtection TRansformer)*** 
    
    최신 기술로, 객체 간의 '관계'를 파악하는 능력이 좋다. 예를 들어 "벽체 선 위에 있는 사각형은 창문일 확률이 높다"는 식의 문맥적 이해를 돕는다.
    

### 시맨틱 vs 인스턴스 세그멘테이션

- 시맨틱 세그멘테이션(Semantic) : 같은 종류는 모두 하나의 덩어리로 본다.
    
    도면 내의 모든 '벽'을 전부 파란색으로 칠한다. 벽 A와 벽 B를 구분하지 않고 그냥 '벽이라는 종류'만 구분한다.
    
- 인스턴스 세그멘테이션(Instance) : 같은 종류라도 각각을 개별로 구분한다.
    
    '방 1', '방 2', '방 3'을 각각 다른 색으로 칠한다. "방이 총 몇 개인가?"를 알아야 하는 물량 산출 프로젝트에서는 인스턴스 방식이 더 유용할 수 있음.
    
1. ***U-Net***
    
    원래 의료용 영상(X-ray 등)에서 장기를 찾아내기 위해 개발된 모델이다.
    
    이미지를 압축했다가 다시 늘리는 구조(U자 모양) 덕분에 아주 미세한 선이나 경계선을 아주 잘 찾아낸다. 도면에서 '벽체 선'만 정확하게 추출하거나, 복잡한 선들 사이에서 '창문 영역'만 분리해낼 때 가장 기본이 되는 강력한 모델.
    
2. ***Mask R-CNN : "박스 치고, 테두리까지 따기"***
    
    앞서 배운 Faster R-CNN(객체 탐지)의 업그레이드 버전이다.
    
    "여기에 문이 있다"라고 네모 박스를 친 다음, 그 박스 안에서 다시 문의 정밀한 실루엣(마스크)을 따낸다. 개수 파악과 면적 계산을 동시에 할 때 유리하다.
    
3. ***SAM (Segment Anything Model)***
    
    Meta(페이스북)에서 만든 최신 AI로, 사전 학습 없이도 이미지의 경계선을 기가 막히게 찾아낸다.
    
    수만 장의 도면을 일일이 학습시키기 어려울 때, SAM을 활용하면 방이나 기둥의 영역을 유연하게 잡아낼 수 있다. 프로젝트 목표인 '다양한 패턴에 대한 안정성'을 확보하는 데 큰 도움이 됨.
    

### 선분 검출

1. ***LSD (Line Segment Detector)***
    
    인접한 픽셀끼리 기울기가 비슷한 것들을 묶어 하나의 선분으로 인식한다. 직선뿐만 아니라 그 직선의 '시작점'과 '끝점'을 아주 정확하게 짚어낸다. 도면을 이미지로 변환했을 때, 벽체 선의 길이를 가장 빠르게 파악할 수 있는 도구임.
    
2. ***LCNN (Line-point Catch Network) : "교차점부터 찾는 AI"***
    
    단순히 선을 찾는 게 아니라, 선과 선이 만나는 '교차점(Junction)'을 먼저 찾고 그 점들을 이어야 할지 말지를 AI가 판단한다. 벽과 벽이 만나는 모서리를 기가 막히게 찾아내므로, 도면의 뼈대(Wireframe)를 추출하는 데 매우 강력함.
    
3.  ***HAWP (Holistic Attraction Field Map) : "전체적인 흐름을 보는 AI"***
    
    이미지의 각 픽셀이 "나는 어떤 선분에 속할 확률이 높은가?"를 계산하여 가장 가까운 선분으로 끌어당긴다. 
    LCNN보다 속도가 빠르고, 복잡하게 얽힌 도면에서도 중요한 구조적 선분을 아주 깔끔하게 뽑아낸다. 현재 도면 분석 AI 연구에서 가장 많이 언급되는 모델 중 하나다.
    

### OCR(Optical Character Recognition, 광학 문자 인식)

도면에는 선과 그림만 있는 것이 아니라, "H-Beam 300x150", "거실", "CH=2400" 같은 텍스트 정보가 가득하다. 아무리 벽을 잘 찾아도 그 벽이 무슨 재질인지, 방의 이름이 무엇인지 읽지 못하면 정확한 견적서를 만들 수 없다. OCR은 도면 속의 글자를 읽어 AI 에이전트의 '눈'을 완성하는 기술이다.

1. ***PaddleOCR: "도면 분석의 우등생"***

현재 산업 현장에서 가장 많이 쓰이는 모델 중 하나다. 글자를 찾는 단계(Detection)와 읽는 단계(Recognition)가 매우 조화롭고, 특히 회전된 글자나 세로로 적힌 글자를 아주 잘 읽는다.
속도도 매우 빨라 수백 장의 도면 처리에 적합하다.

1. ***EasyOCR: "빠르고 간편한 범용 도구"***

파이썬 환경에서 가장 쉽게 설치하고 테스트해 볼 수 있는 모델이다. 80개 이상의 언어를 지원하며, 딥러닝 프레임워크인 PyTorch 기반이라 커스터마이징이 쉽다.

프로젝트 초기 단계에서 빠르게 프로토타입을 만들 때 좋다. 한글 인식 성능도 준수하여 "거실", "주차장" 같은 한글 텍스트를 읽는 데 유용하다.

1. ***TrOCR: "최신 기술(Transformer)의 정수"***

기존 방식과 달리 이미지 자체를 문장으로 이해하는 '트랜스포머' 구조를 사용한다. 선이나 기호가 글자와 겹쳐 있어도 문맥을 통해 글자를 유추하는 능력이 뛰어나다.

도면 선 위에 글자가 겹쳐 써진 경우, 일반적인 OCR은 선을 글자로 착각해 오류가 나기 쉬운데, TrOCR은 이런 노이즈 상황에서 훨씬 정확하게 잡아낸다. 높은 정확도(오차율 5% 미만)를 위해 최종 단계에서 사용하기 좋음.

### GNN(Graph Neural Networks)

기존의 AI가 도면을 픽셀(점)의 집합으로 봤다면, GNN은 도면을 점(Node)과 선(Edge)의 연결 관계로 본다.

- 노드(Node) : 선분(Line Segment)이나 교차점(Junction)이 노드가 된다.
- 엣지(Edge) : 두 선분이 연결되어 있는지, 혹은 벽과 벽이 만나 방을 형성하는지의 '관계'가 엣지가 된다.

**GNN의 장점**

1. 단순히 "선이 여기 있다"가 아니라, "선 A, B, C, D가 서로 연결되어 닫힌 공간을 만들었으니 이것은 '방'이다"라는 논리적 추론이 가능하다.
2. 설계사마다 그리는 스타일이 달라도 '방은 벽으로 둘러싸여 있다'는 연결의 법칙은 변하지 않는다. 법칙에 의해 판단하기 때문에 다양한 도면 패턴에도 안정적인 정확도를 낸다.
3. "이 벽은 거실에 속해 있고, 거실은 101동에 속해 있다"는 식의 포함 관계를 파악하는 데 적합하다.

### 벡터 그래픽 직접 처리

실제 CAD 파일은 좌표 데이터인 '벡터(Vector)' 방식이다. 이를 이미지로 변환하지 않고 숫자 데이터 그대로 AI에 넣어버리는 방식을 말한다.

**장점**

1. 이미지는 확대하면 깨지지만, 벡터는 좌표값($x, y$)이라 아무리 작은 부품(H-Beam의 상세 규격 등)도 오차 없이 읽을 수 있다. 오차율 5% 미만 달성에 결정적임.
2. "벽체 레이어만 읽어와"라는 명령이 코드 한 줄로 가능하다. 이미지 분석에서 벽체만 골라내려고 고생할 필요가 없다.
3. 선(Line), 원(Circle), 폴리라인(Polyline) 같은 CAD의 기본 단위를 AI가 직접 인식하므로, 물량 산출 속도가 '초 단위'로 빨라진다.

**참고할 만한 논문/연구 흐름**

- *Automated Wall Detection in 2D CAD Drawings* (Wolf et al.)
- *Residential Floor Plan Recognition and Reconstruction* (Alibaba)
- *Deep Floor Plan Analysis for Complicated Drawings Based on Style Transfer*
- 검색 키워드 : cad recognition, construction drawing recognition, floor plan parsing

## 건축 관련 LLM/NLP 필수 개념 (문제정의 2 관련)

건축 현장에서는 같은 자재를 두고도 설계사, 시공사, 반장님마다 부르는 이름이 제각각이다. 이 "말이 안 통하는 문제"를 AI가 어떻게 해결하는지, 임베딩과 벡터 DB의 관점에서 설명해 보려고 한다.

### 텍스트 임베딩

"강관 덕트"와 "STEEL DUCT"는 글자(텍스트)만 보면 완전히 다르지만, 임베딩 모델을 거치면 숫자 좌표상에서 매우 가까운 위치에 놓이게 된다.

1. ***Sentence-BERT***
    
    문장 전체의 맥락을 파악하는 데 특화된 모델이다. 단어 하나하나의 뜻보다는 "이 문장이 결국 무엇을 말하는가?"를 수치화한다.
    
2. ***KoSimCSE***
    
    한국어의 문장 유사도를 정교하게 파악하도록 학습된 모델이다.
    
3. ***BGE-M3***
    
    다국어 지원이 강력하며, 짧은 단어부터 긴 문서까지 성능이 안정적인 로컬 모델이다. "H형강"과 "H-Beam"처럼 언어가 섞인 데이터를 매핑할 때 유리하다.
    
4. ***OpenAI 임베딩***
    
    OpenAI와 Cohere 임베딩은 사용자가 직접 복잡한 AI 모델을 설치하거나 학습시킬 필요 없이, API(인터넷 연결)를 통해 고성능 임베딩 기능을 빌려 쓰는 서비스다. 
    
    다만 보안 위험이 있어 실제 프로젝트에서는 OpenAI로 빠르게 로직을 검증해보고, 보안이 중요한 핵심 데이터는 한국어 특화 모델(BGE-M3 등)을 서버에 직접 띄워 사용하는 '하이브리드' 방식을 추천함
    
    - **text-embedding-3-small/large**
        
        전 세계에서 가장 많이 쓰이는 임베딩 모델 중 하나다. 수조 개의 문장을 학습한 GPT 기반의 모델이라, 단어 사이의 숨은 의미를 아주 잘 파악한다.
        
        **장점**
        
        1. 영어와 한국어를 넘나드는 능력이 탁월하다. "H-Beam"과 "H형강"이 같은 의미라는 것을 별도의 가르침 없이도 잘 알고 있다.
        2. 임베딩 결과물(숫자 뭉치)의 길이를 조절할 수 있어, 저장 공간(벡터 DB)을 효율적으로 쓸 수 있다.
5. ***Cohere 임베딩***
    - **embed-multilingual-v3.0**
        
        기업용(Enterprise) 검색과 실무 데이터 처리에 특화된 모델을 제공한다.
        
        **장점**
        
        1. '다국어 검색'에 미친 듯이 진심인 모델이다. 100개 이상의 언어를 지원하며, 특히 검색 정확도가 높기로 유명함.
        2. 벡터 데이터를 아주 작게 압축해서 저장할 수 있는 기술이 있어, 대규모 견적 데이터를 처리할 때 비용을 아낄 수 있다.
        3. 단순한 유사도가 아니라, '질문과 답변'의 관계를 더 잘 이해하도록 설계되었다.
        4. 제안서에 나온 것처럼 "비슷한 공사 현장에서 함께 사용되는 자재들을 그룹핑"하는 복잡한 맥락 분석에 유리하다.

### 벡터 데이터베이스 & 유사도 검색

임베딩을 통해 모든 자재 이름을 숫자로 바꿨다면, 이제 수만 개의 과거 단가 데이터 중에서 가장 유사한 것을 광속으로 찾아내야 한다. 이때 사용하는 것이 벡터 DB다.

1. ***FAISS***
    
    Facebook(Meta)에서 만든 라이브러리로, 수억 개의 데이터 중에서도 유사한 것을 찾는 속도가 매우 빠르다. (로컬 서버 설치형)
    
2. ***Qdrant / Pinecone***
    
    클라우드 기반의 벡터 DB로, 단순히 검색만 하는 게 아니라 데이터를 관리하고 업데이트하기 편리하다.
    

### **RAG(Retrieval-Augmented Generation)**

과거 견적서를 검색해 LLM이 새 견적을 생성/추천하는 데 쓰인다.

1. "이걸 뭐라고 불렀더라?" - 용어 파편화 해결
    
    제안서의 문제정의 2와 직결되는 부분이다. 설계사마다 "강관 덕트", "STEEL DUCT", "S.D" 등 제각각으로 쓴 용어들을 RAG가 과거 매핑 기록을 근거로 "이건 같은 자재입니다"라고 판단하여 일관된 단가를 매핑해 준다.
    
2. "그때 얼마에 계약했지?" - 실시간 단가 업데이트
    
    건설 자재비는 변동이 심하다. LLM을 매번 다시 학습시킬 수는 없지만, RAG를 이용하면 어제 업데이트한 엑셀 견적서 파일만 벡터 DB에 넣어두어도 AI가 오늘 즉시 그 가격을 반영해 견적을 추천한다.
    
3. "왜 이 가격이야?" - 근거 있는 의사결정 지원
    
    엔지니어의 가장 큰 피로도는 "수치가 틀리면 어떡하지?"라는 불안감이다. RAG는 답변할 때 "과거 OO 프로젝트 견적서 15페이지를 참고했습니다"라고 출처를 제시할 수 있다.
    

### Entity Normalization와 Record Linkage

쉽게 말해 이름이 제각각인 데이터를 하나의 이름으로 통일하고, 서로 흩어진 정보를 하나로 묶는 과정이다.

***Entity Normalization***

데이터를 분석하기 좋게 '표준 형식'으로 맞추는 작업이다.

- 형식 통일 : 대소문자 변환(H-beam → H-BEAM), 공백 제거, 특수문자 처리
- 단위 변환 : "1000mm"와 "1m"를 같은 수치 데이터로 정규화함
- 대표어 매핑 : 다양한 이명을 하나의 표준어(Canonical Name)로 바꿈

***Record Linkage***

서로 다른 데이터셋에서 실제로 같은 대상을 가리키는 레코드를 찾아 연결하는 기법이다. 도면(이미지/벡터)에서 찾은 '기둥 좌표' 데이터와 견적서(엑셀)에 적힌 '단가' 데이터를 서로 연결해야 "이 기둥 하나에 얼마"라는 최종 결과가 나온다.

- 결정적 매핑(Deterministic) : ID나 코드처럼 고유한 값이 일치할 때 바로 연결한다. (예: 자재 코드 'A-01'이 양쪽에 있을 때)
- 확률적 매핑(Probabilistic) : 고유 ID가 없을 때 사용한다. 이름의 유사도, 규격, 위치 정보를 종합하여 "이 두 데이터는 동일할 확률이 95%다"라고 판단하여 연결한다.

### 클러스터링(Clustering)

클러스터링은 데이터들 사이의 유사성을 계산하여 정답(Label)이 없는 상태에서 비슷한 것들끼리 무리(Cluster)를 지어주는 기술이다.

1. ***K-means***
    
    데이터들을 K개의 묶음으로 나누는 방식으로, 미리 정해진 K개의 중심점을 찍고, 각 데이터가 가장 가까운 중심점으로 모이면서 그룹을 형성한다.
    
    속도가 매우 빠르고 결과가 직관적이다. 그룹의 개수(K)를 사람이 미리 정해야 하며, 데이터가 동그란 모양이 아닐 경우 성능이 떨어진다.
    
2. ***계층적 클러스터링 (Hierarchical Clustering)***
    
    데이터들을 하나씩 묶어가며 나무 모양의 구조(Dendrogram)를 만드는 방식이다. 가장 가까운 두 데이터를 먼저 묶고, 그 묶음과 또 가까운 데이터를 계속 묶어 올라간다.
    
    그룹의 개수를 미리 정할 필요가 없으며, 데이터 간의 '족보'를 확인할 수 있다.
    
    자재 분류 체계를 만들 때 유리하다. '나사'와 '못'을 먼저 묶고, 이를 다시 '철물'이라는 큰 범주로 묶는 과정을 시각적으로 파악할 수 있다.
    
3. ***HDBSCAN***
    
    데이터가 얼마나 빽빽하게 모여 있는지를 기준으로 그룹을 찾는다. 밀도가 높은 부분은 그룹으로 인정하고, 너무 멀리 떨어져 있는 데이터는 '노이즈(이상치)'로 과감히 버린다.
    
    그룹의 개수를 스스로 찾아내며, 모양이 불규칙한 데이터도 아주 잘 찾아낸다. 
    "비슷한 자재들끼리 묶어봐"라고 명령하면, AI가 스스로 판단하여 어떤 그룹은 크게, 어떤 그룹은 작게 나누며, 아예 생소한 자재는 "분류 불가(노이즈)"로 따로 빼준다.

이 모습은 건설파트너 제안서의 다음 목표와 거의 일치한다.

- "오차율 5% 이하"

3. 도메인-기술 결합이 결정적이다. 포비콘이 빠르게 99%를 달성한 비결은 컴퓨터 비전 단독이 아니라 건설 도메인 지식과 비전 AI의 결합이다. 수강생으로서 "왜 이 자재가 이 위치에 있어야 하는가" 같은 도메인 규칙을 모델 설계에 녹이는 능력이, 단순히 YOLO를 잘 돌리는 것보다 훨씬 큰 차이를 만든다.

***1. "Symbol Spotting on Digital Architectural Floor Plans Using a Deep Learning-based Framework" (arXiv 2006.00684)***

"대형 도면에서 작은 심볼을 놓치지 않는 전략"

- 핵심 내용: 거대한 건축 도면 이미지에서 아주 작은 크기의 심볼(문, 창문 등)을 검출하기 위해 YOLO 프레임워크를 최적화한 연구입니다.
- 기술적 특징: 고해상도 도면을 그대로 모델에 넣으면 심볼이 너무 작아져 인식률이 떨어지는 문제를 해결하기 위해, 이미지를 일정한 크기로 자르는 '타일 기반(Tiling)' 학습 및 추론 전략을 제안합니다.
- 프로젝트 적용점: 우리 도면 역시 캔버스는 넓고 부호(MC1, SG2 등)는 작기 때문에, YOLOv8/v11 모델에 타일링 기법을 적용하여 zero-shot 검출 성능을 가장 먼저 확인해볼 수 있는 '빠른 실행형' 모델입니다.
- 관련 오픈소스: [https://github.com/sanatladkat/floor-plan-object-detection](https://github.com/sanatladkat/floor-plan-object-detection) (YOLOv8로 기둥/벽/문/창 검출+카운팅, Streamlit 데모까지 됨)

***2. "GroundCount: Grounding Vision-Language Models with Object Detection for Mitigating Counting Hallucinations" (arXiv 2603.10978)***

"VLM의 고질적인 문제인 '숫자 세기 환각' 치료제"

- 핵심 내용: VLM(시각-언어 모델)이 이미지를 보고 개수를 잘못 세는 '환각(Hallucination)' 현상을 해결하기 위해, 객체 검출 모델(ODM)을 비서처럼 활용하는 프레임워크입니다.
- 기술적 특징: VLM에게 이미지만 주는 것이 아니라, 검출기가 찾아낸 좌표(Bounding Box) 데이터를 텍스트 프롬프트로 함께 전달합니다. 이를 통해 정확도를 81.3%까지 끌어올렸습니다.
- 프로젝트 적용점: "VLM 단독으로는 카운팅이 어렵다"는 사실을 인정하고 시작하는 전략입니다. 모델을 직접 수정하지 않고, 검출 결과를 프롬프트에 포함하는 방식(Plan A)으로 즉시 실험이 가능합니다.

***3. "VLM-Grounder: A VLM Agent for Zero-Shot 3D Visual Grounding" (arXiv 2410.13860)***

"에이전트 방식으로 똑똑하게 찾아내는 Zero-shot 탐색기"

- 핵심 내용: 3D 공간에서 VLM이 에이전트 역할을 수행하며, 검출기(GroundingDINO, SAM)와 협업하여 특정 객체를 추적하고 위치를 파악하는 프레임워크입니다.
- 기술적 특징: 단순히 한 번 보고 끝내는 것이 아니라, 여러 뷰(View)를 결합하고 피드백 루프를 통해 정밀한 바운딩 박스를 추정합니다.
- 프로젝트 적용점: 도면 분석을 일종의 에이전트 태스크로 접근합니다. "도면에서 H-beam 기호를 찾아줘"라는 자연어 명령을 Grounding DINO로 실행하고, VLM이 그 결과를 최종 검증하는 '고급 zero-shot 파이프라인' 구축에 적합합니다.

***4. LISA++***

LISA++ 같은 reasoning segmentation을 도면 이미지에 적용하면 "도면에서 부재 일람표 영역을 찾아 분할해줘" 같은 자연어 명령으로 표 영역 마스크를 받을 수 있다. 이건 룰 기반 DBSCAN보다 일반화 가능성이 큼.

***5. X-SAM***

X-SAM의 VGD(시각적 그라운딩 분할) 기능은 우리가 가려던 "박스 그리기 + AI 검색"(Togal식 사람-AI 협업) UX와 거의 정확히 맞아 들어간다. **사용자가 도면에서 H-Beam 부호 하나에 박스 치면 → 도면 전체에서 같은 인스턴스 다 찾아 분할 → 카운트, 이런 흐름의 방식**을 쓸 수 있다.

- **사람-AI 협업 — "박스 그리기 + AI 검색"**
    
    이게 학생 프로젝트가 가장 빨리 도달할 수 있는 UX. 사용자가 도면에서 H-BEAM 부호 하나에 박스를 그리면, 그 위치의 ezdxf 엔티티를 가져와서 비슷한 패턴(같은 텍스트, 같은 블록, 같은 레이어)을 도면 전체에서 찾아 카운트함. 룰 기반으로 충분히 구현 가능하다.
    
    이 패턴은 한국 회사별 부호 다양성 문제를 우아하게 우회한다. AI에게 "MC1이 뭔지 알아내라"고 시키지 않고, 사용자가 한 번 알려주는 것.

        도면을 잘 그렸다면 RG1은 보통 블록(Block) 으로 저장되어 있다. 한 번 정의해두고 도면에 100번 찍으면, 파일에는 "RG1 블록을 (x, y) 위치에 100번 찍었다"라고 기록된다. 이걸 코드로 구현해서 개수 계산하면 됨.
        
        - 또는 그냥 선으로 그려진 경우엔 선의 길이를 합산한다. ezdxf가 선의 시작점과 끝점 좌표를 알려주니까, 피타고라스 정리로 길이 계산이 가능함
            
            ```python
            import math
            
            steel_total_length = 0
            for e in msp:
                if e.dxftype() == 'LINE' and e.dxf.layer == 'STEEL':
                    # 시작점, 끝점 좌표
                    x1, y1, _ = e.dxf.start
                    x2, y2, _ = e.dxf.end
                    # 두 점 사이 거리 계산
                    length = math.sqrt((x2-x1)**2 + (y2-y1)**2)
                    steel_total_length += length
            
            print(f"STEEL 레이어 총 선 길이: {steel_total_length:.2f} mm")
            print(f"= {steel_total_length/1000:.2f} m")
            ```

### 부재별 카운팅 규칙이 다른 것에 대한 문제 해결

사실 건설 도면 자동화의 가장 어려운 문제다. 

**평면도에서 부호 위치 파악 → 일람표에서 해당 부호의 규격 조회 → 물량 계산**
처음엔 개수만 맞히고, 그 다음에 규격 매칭, 그 다음에 길이, 이렇게 하나씩 추가로 성능을 높여보자.

- 같은 부호라도 크기가 다른 부재들이 있으므로, 유의해서 수량 계산해야 한다.
    - 1F 도면: MC1 10개 × (600×600mm 단면, 철근 8개, 층고 3.5m)
    - 2F 도면: MC1 10개 × (500×500mm 단면, 철근 6개, 층고 3.2m)
    - 3F 도면: MC1 10개 × (400×400mm 단면, 철근 4개, 층고 3.0m)
- 층별로 부재 개수를 세는 게 맞다.
- 아래 내용처럼 부재별 카운팅이 다 다르다.
    
    
    | 부호 표시 | 실제 의미 | 카운팅 규칙 |
    | --- | --- | --- |
    | 평면도에 `MC1` 텍스트 1개 | 그 위치에 기둥 1개 | 1대1 매칭 |
    | 평면도에 `MC1` 그룹 1개 + "x10" 표기 | 같은 기둥 10개 일괄 표시 | 1대N 곱셈 |
    | 일람표에 `MC1` 행 1개 | 부재 정의 (실제 개수와 무관) | 카운트 안 함 |
    | 평면도에 `D25@200` 표기 | 200mm 간격으로 D25 철근 배치 | 길이 / 간격 = 개수 |
    | 일람표 "수량 = 24" 셀 | 명시적 수량 | 그 숫자 그대로 |
    | 보 일람표 `SB0` 행 + 평면도 `SB0` 위치 8군데 | 보 8개, 일람표는 정의용 | 평면도 카운트만 사용 |
1. **사람(도메인 전문가)이 카운팅 규칙을 규칙 카탈로그로 정의해둔다.**
5~10개 정도의 규칙을 사람이 짜야 한다. 도메인 전문가(건설사 견적팀)와 함께 정의하는 게 핵심이고, 이걸 외부에서 쉽게 만들어주는 도구는 없다.
    
    규칙 카탈로그 : "부재별로 어떻게 세야 하는지 사람이 미리 정의한 규칙들"
    
    - 규칙 1: 평면도에 부호 1개 = 그 위치에 부재 1개 (1대1 매칭)
    - 규칙 2: 일람표에 있는 부호는 정의용 → 카운트 제외
    - 규칙 3: `MC1 x10` 같은 곱셈 표기 → 곱하기 처리
    - 규칙 4: 일람표 "수량 = 24" 셀 → 그 숫자 그대로
    - 예시)
        
        ```python
        # 규칙 1: 1대1 매칭 (평면도 부호 위치 = 개수)
        def count_one_to_one(dxf, drawing_id, symbol):
            locations = find_text_locations(dxf, symbol, drawing_id)
            return len(locations)
        
        # 규칙 2: 일람표 명시 수량 사용
        def count_from_schedule(dxf, drawing_id, symbol):
            # 일람표 영역에서 해당 부호 옆 "수량" 셀 추출
            quantity_cell = parse_schedule_cell(dxf, drawing_id, symbol, column="수량")
            return int(quantity_cell)
        
        # 규칙 3: 철근 배근 간격 계산 (D25@200)
        def count_rebar_by_spacing(dxf, drawing_id, length_mm, spacing_mm):
            return int(length_mm / spacing_mm) + 1
        
        # 규칙 4: "x10" 같은 곱셈 표기 처리
        def count_with_multiplier(dxf, drawing_id, symbol):
            base_count = count_one_to_one(dxf, drawing_id, symbol)
            multipliers = find_multiplier_text_near(dxf, symbol)  # "x10", "TYP" 등
            return base_count * (multipliers if multipliers else 1)
        ```
        
    - 단위 중량 표 만들기 예시)
        
        Togal이 AIA 표준을 따르듯, 우리는 KS D 3503 일반구조용 압연강재 / KS D 3530 일반구조용 경량형강 같은 한국 KS 표준의 단위 중량 표를 매핑 카탈로그로 만든다.

2. **dxf로부터 받은 정보를 가지고 LLM은 "어떤 규칙을 적용할지"만 결정한다.**

### 실제 구현 방법

#### **1. 룰 기반 + AI 하이브리드**

**ezdxf 파싱 결과를 잘 구조화해서 LLM에 전달해야 한다.** 즉, llm이 잘 판단할 수 있도록 수월한 재료를 준비해줘야 한다는 뜻. dxf 파일 안에서 뽑아낼 수 있는 그라운딩 정보를 룰 기반으로 뽑아내서, ai에게 먹이면 ai가 정보를 보고 사람처럼 판단하고 추측해서 결과를 냄.

- 룰 기반은 "이 영역의 모양은 격자다, 텍스트 밀도 높다, 헤더 단어 있다" 같은 물리적 특성 측정까지가 한계다.
- "이 영역이 일람표인지 평면도인지"라는 의미 판단은 LLM이 해야 한다.

DBSCAN 클러스터링, bbox 계산, 텍스트 카운팅, 레이어 분포 등을 사용해서 llm에게 먹임.
이 모든 것은 수학 공식과 라이브러리 함수로 끝난다. 모델 학습도 필요 없고, 구현하는 코드도 짧다.

- **LLM이 정보를 받았을 때 가장 잘 판단하는 형태**
    1. **의미 단위로 묶여있어야 함**
        
        "TEXT 타입이 1324개"보다 "부재 부호가 35개, 일반 라벨이 286개"가 더 유용하다. 분류 기준이 도메인적이어야 LLM이 판단 근거로 쓴다.
        
    2. **비교 가능한 형태로 나와야 함**
        
        절대값보다 비율, 단일 숫자보다 분포가 LLM 판단에 도움된다. "MC1 35회"만 있으면 그게 많은 건지 적은 건지 모르지만, "전체 부재 87개 중 MC1이 40%"면 즉시 의미가 잡힌다.
        
    3. **위계가 있어야 함**
        
        평평한 리스트보다 "도면 전체 → 영역별 → 영역 안의 부호별" 같은 계층 구조가 LLM의 추론을 돕는다.
        
    4. **도메인 힌트가 함께 있어야 함**
        
        숫자만 주면 LLM이 의미 추론에 환각 위험이 있지만, "이건 일람표 후보, 저건 평면도 후보"라는 코드 기반 후보 라벨을 함께 주면 LLM은 확인만 하면 된다.

- 코드 예시)
    
    ```python
    import ezdxf
    import numpy as np
    from sklearn.cluster import DBSCAN
    from collections import Counter
    
    def extract_grounding_info(dxf_path):
        """LLM에 줄 그라운딩 정보를 통째로 만든다 — 룰 기반 100%"""
        
        # 1단계: ezdxf로 좌표 포함 텍스트 추출
        doc = ezdxf.readfile(dxf_path)
        msp = doc.modelspace()
        
        texts = []
        for e in msp:
            if e.dxftype() in ("TEXT", "MTEXT"):
                texts.append({
                    "text": e.dxf.text.strip(),
                    "x": e.dxf.insert.x,
                    "y": e.dxf.insert.y,
                    "layer": e.dxf.layer,
                })
        
        # 2단계: 좌표 기반 DBSCAN 군집화
        coords = np.array([(t["x"], t["y"]) for t in texts])
        labels = DBSCAN(eps=500, min_samples=5).fit_predict(coords)
        
        # 3단계: 군집별 정보 집계
        clusters = []
        for cid in set(labels):
            if cid == -1:  # 노이즈는 제외
                continue
            members = [texts[i] for i, l in enumerate(labels) if l == cid]
            xs = [m["x"] for m in members]
            ys = [m["y"] for m in members]
            layers = Counter(m["layer"] for m in members)
            sample_texts = [m["text"] for m in members[:5]]
            
            clusters.append({
                "cluster_id": int(cid),
                "bbox": (min(xs), min(ys), max(xs), max(ys)),
                "text_count": len(members),
                "dominant_layer": layers.most_common(1)[0][0],
                "layer_distribution": dict(layers),
                "sample_texts": sample_texts,
            })
        
        # 4단계: 부호 카운팅 (전체 도면 기준)
        symbol_counts = Counter(t["text"] for t in texts if t["text"])
        
        return {
            "total_texts": len(texts),
            "clusters": clusters,
            "top_symbols": dict(symbol_counts.most_common(20)),
        }
    
    # 실행
    info = extract_grounding_info("도면1.dxf")
    ```

### 우리 프로젝트에 도구가 들어갈 세 자리

먼저 우리가 다루는 정보를 세 종류로 나눠보면:

1. 선과 면 (벽선, H-Beam 윤곽, 표 테두리, 방 영역) → 기하 정보
2. 텍스트 (MC1, H-400×200, 기초구조도) → 문자 정보
3. 텍스트 + 위치의 결합 의미 (이 텍스트는 표 헤더, 저 텍스트는 셀) → 레이아웃 의미

| 자리 | 도구 | 학습 필요? | 우리 프로젝트 역할 |
| --- | --- | --- | --- |
| 기하 정보 | ezdxf | 학습 0 | 메인 — 벡터 데이터에서 직접 좌표 추출 |
| 기하 정보 (백업) | OpenCV LSD / findContours | 학습 0 | 백업 — 벡터 망가진 케이스 |
| 기하 연산 | Shapely | 학습 0 | 메인 — 폐곡선·교차·면적 계산 |
| 문자 정보 | PaddleOCR / EasyOCR | 사전학습됨 (우리 추가학습 X) | 백업 — DXF에 텍스트 누락 시 |
| 레이아웃 의미 | PP-Structure | 사전학습됨 (우리 추가학습 X) | 보조 — 일람표 행·열 구조화 |
| 레이아웃 의미 | LayoutLMv3 | 미세조정 권장 | 장기 후보 — 라벨 쌓인 뒤 |
| 문서 → JSON | Donut / Pix2Struct | 미세조정 권장 | 장기 후보 |

### LLM의 부족함을 어떻게 채워주는지 한눈에

| LLM의 약점 | 채워주는 도구 | 어떻게 |
| --- | --- | --- |
| 카운팅 환각 | ezdxf + Counter | 코드가 미리 카운트해서 LLM에 숫자만 넘김 |
| 좌표→시각 군집 인식 못 함 | DBSCAN + Shapely | 좌표를 미리 영역·다각형으로 변환 |
| 폐곡선·교차 추론 약함 | Shapely | 닫힌 영역·교차점을 결정론적으로 계산 |
| 표 행·열 구조 이해 약함 | PaddleOCR PP-Structure | 격자 시각 패턴으로 표 구조 추출 |
| 텍스트와 인접 객체 매칭 약함 | (장기) LayoutLMv3 | 공간 관계 학습된 토큰 분류 |
| 이미지 안 보고 좌표만 봄 | PaddleOCR (백업) | 이미지에서 텍스트 직접 인식 |

### 한계점 정리

1. 모든 외부 모델의 공통 한계
    
    학습 분포가 한국 건설도면과 멀어요. PaddleOCR은 일반 한글 OCR 데이터로 학습됐고, PP-Structure는 사무 문서·재무 보고서 표로 학습됐고, LayoutLM은 영문 양식 문서로 학습됐어요. *"한국 건설사 구조도면의 일람표"*라는 특수 도메인은 어디에도 없어요.
    
2. 그래서 zero-shot으로 쓸 수 있는 건 80% 수준의 1차 판정·1차 추출이고, 그걸 그대로 카운팅·견적에 쓰면 5% 오차 목표 달성 못 해요. 반드시 ezdxf 결과와의 교차 검증이 따라붙어야 해요.
3. 고전 CV 알고리즘들(LSD, findContours, Hough)의 한계는 다른 결이에요. 결정론적이라 환각은 없지만, 도면 품질에 민감해요. 선이 흐릿하거나 끊겨있으면 LSD가 한 선분을 두 개로 쪼개고, 모폴로지 연산 파라미터 튜닝이 도면마다 다르게 필요해요. 즉, **"학습 없음"의 대가는 "도면마다 튜닝"**이에요.
4. Shapely는 한계가 거의 없어요. 좌표만 정확하면 결과도 정확해요. 다만 좌표 자체의 정확도(ezdxf 정확도)에 종속돼요.
5. 우리 LLM 백본 자체의 한계도 변하지 않아요. 위 도구들로 입력 품질을 끌어올려도, LLM이 마지막 라우팅·해석 단계에서 환각을 일으킬 가능성은 남아요. 이건 self-verification, multi-agent 패턴으로 따로 대응할 일이에요.

## 구체적으로 계획 잡기

- **팀웍 주의사항**
    - 각 단계는 이전 단계 대비 **"무엇이 얼마나 좋아졌는지"** 숫자로 보여줘야 한다. 막 추가하지 않고, 이전 버전의 오차 분석이 다음 버전의 설계 근거가 되도록.
    - 팀별 역할 분담 및 진행에 있어 중요하게 고려할 점
        - 매주 정해진 요일(월 or 금 추천)에 **"이번 주 금요일에 무엇이 끝나 있을 것인가"**를 한 문장으로 합의하자.
            - 예시) "5/18 일요일 18시까지 도면1.dxf의 부호별 카운트 JSON + 오차율 리포트 1장이 깃허브에 올라와 있다”
        - 부족하지만 작동하는 게 중요하다. 완벽하게 하려다가 시간 딜레이시키지 말자.
        - 각자 팀원들의 코드 합치는 **통합 리드** 한명 세우자. 그 리드는 **그 주의 결과물 리포트 작성**해야 한다.
            - 코드/모듈/인터페이스 형식 통일시키도록 사전 협의 필수. 30분~1시간 내로 정확하게 얘기 끝내기.
                - 예시)
                    
                    ```python
                    # 모듈 간 합의: 도면1 처리 결과는 이 형식이어야 한다
                    {
                      "drawing_id": "도면1",
                      "entities": [
                        {"type": "TEXT", "content": "MC1", "x": 1234.5, "y": 678.9, "layer": "TXT"},
                        {"type": "INSERT", "block": "RG1", "x": 2345.6, "y": 789.0, "layer": "SYM"},
                        ...
                      ],
                      "regions": [
                        {"region_id": "A", "bbox": [5234, 3120, 7480, 4530], "type_guess": "table"},
                        ...
                      ],
                      "counts": {
                        "MC1": 24,
                        "MC2": 18,
                        ...
                      }
                    }
                    ```
                    
    - 큰 틀 안에서 해야 할 일 (1단계에 할 일은 따로 작성함)
        - LLM 프롬프트 설계
        - LLM 라우팅 로직 (어떤 영역에 어떤 룰을 적용할지)
        - 평가 지표·오차율 측정 코드
        - 멘토님 커뮤니케이션 담당
        - 부재 카운팅 규칙 카탈로그 작성 (멘토님과 협업)
    - 인터페이스란?
        
        두 사람이 일을 주고받을 때 미리 합의한 약속. "무엇을 어떤 형태로 주고받을지"를 미리 정해둔 약속.
        
        A가 "썬 김치"를 B에게 넘기는데, B는 "통김치 한 포기 통째로"가 올 줄 알았음. A가 "잘게 다진 김치"를 가져왔고, B는 "이걸로는 내가 짠 레시피 안 맞는데?"하면서 다시 손질해야 한다. 시간 허비.
        
        이와 같이 멤버 A가 만든 모듈의 결과를 멤버 B의 모듈이 받아서 처리한다. "A가 무엇을 어떤 형태로 출력하는가"와 "B가 무엇을 어떤 형태로 받기를 기대하는가"가 같아야 둘이 합쳐진다.
        
        예시) "A는 DXF 파일 경로를 받아서 엔티티 리스트를 JSON으로 내놓는다. B는 그 JSON을 받아서 부호만 골라낸 JSON을 내놓는다..."
