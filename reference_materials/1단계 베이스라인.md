### 앱 작동방식

DXF 파일을 PNG 이미지로 그리면서 "이 PNG의 어느 픽셀이 DXF의 어느 좌표인지"를 기억해뒀다가, 사용자가 픽셀로 박스를 그리면 그걸 DXF 좌표로 거꾸로 환산해서 그 영역 안의 텍스트만 골라낸다.

### 1단계: DXF 파일 안에는 무엇이 있나

`도면1.dxf` 파일을 ezdxf로 열면, 그 안의 모든 객체(LINE, TEXT, INSERT 등)는 **DXF 좌표**라는 숫자 좌표 위에 있어요.

예를 들어 "VG1"이라는 텍스트 하나는 이런 정보를 가지고 있어요:

`타입: TEXT
내용: "VG1"
위치: x=296706.5, y=1206368.9
레이어: TEX`

도면1.dxf 안에는 이런 텍스트가 1,324개 있고, 각각 자기 (x, y) 좌표를 갖고 있어요.

**중요한 사실**: DXF의 (x, y) 좌표는 **실세계 단위(mm)**예요. 즉 x=296706이라는 건 도면 원점에서 296,706mm = 약 296미터 떨어져 있다는 뜻. (도면이 크지만 1:100 축척으로 그려진 거니까 실제 건물은 약 30m 가량)

그리고 도면1.dxf 전체의 좌표 범위는:

- x: 30,850 ~ 427,324 (왼쪽 끝~오른쪽 끝)
- y: 1,067,924 ~ 1,217,613 (아래쪽 끝~위쪽 끝)

이 사각형 안에 모든 도면 요소가 들어있어요.

이렇게 만들어진 PNG는 **픽셀 좌표**를 가져요. 좌상단이 (0, 0), 우하단이 (1600, 1200).

지금 우리 앱에서는 화면에 맞게 985×369 픽셀로 리사이즈해서 보여주고 있어요(아까 스크린샷의 디버그 정보 기억하시죠?).

**핵심 포인트**: 이때 우리는 **두 좌표계 사이의 매핑 관계**를 알고 있어요.

`PNG의 (0, 0) 좌상단     ↔  DXF의 (30850.3, 1217613.3) 좌상단
PNG의 (985, 369) 우하단 ↔  DXF의 (427324.1, 1067924.3) 우하단`

PNG 전체가 DXF 전체 영역을 정확히 덮고 있어요. 비율도 유지된 채로.

### 3단계: 사용자가 박스를 그릴 때

`streamlit-drawable-canvas`가 마법을 부려요. 사용자가 마우스로 박스를 그리면, 이 라이브러리가 JSON으로 박스 정보를 돌려줘요:

json

`{

  "top": 8,
  "width": 103,
  "height": 79
}`

이건 PNG 위에서 픽셀 좌표예요. **"PNG의 (638, 8)에서 시작해서 가로 103픽셀, 세로 79픽셀의 박스를 그렸다"** 는 뜻이에요.

---

### 4단계: 픽셀 좌표를 DXF 좌표로 변환

이게 핵심이에요. **2단계에서 알고 있는 매핑 관계**를 이용해서 비례식을 풀어요:

`픽셀 박스 좌상단: (638, 8)
픽셀 박스 우하단: (638+103, 8+79) = (741, 87)

PNG 전체 크기: 985 × 369 픽셀
DXF 전체 크기: x폭 396,473.8 / y폭 149,689.0

변환 공식:
  dxf_x = dxf_xmin + (px / png_w) × dxf_x폭
  dxf_y = dxf_ymax - (py / png_h) × dxf_y폭   ← y는 뒤집힘`

왜 y가 뒤집히냐? **PNG는 위쪽이 y=0인데, DXF는 아래쪽이 y=0이에요.** 컴퓨터 화면은 위에서 아래로 픽셀을 세지만, 수학/도면은 아래에서 위로 세요. 그래서 y만 뒤집어줘야 해요.

이 비례식을 풀면:

`픽셀 (638, 8)   →  DXF (287652.6, 1214368.0)
픽셀 (741, 87)  →  DXF (329111.3, 1182320.7)`

(아까 디버그 정보에 나온 값과 정확히 일치해요!)

즉 **사용자가 친 박스는 DXF 좌표로 (287652.6, 1182320.7) ~ (329111.3, 1214368.0) 영역**이에요.

---

### 5단계: 그 영역 안의 부재 부호 카운트

이제 ezdxf로 다시 한번 DXF 파일을 열고, 모든 TEXT/MTEXT를 훑으면서:

python

`for e in msp:
    if e.dxftype() in ["TEXT", "MTEXT"]:
        content = e.dxf.text.strip()  # 예: "VG1"
        x, y = e.dxf.insert.x, e.dxf.insert.y
        
        # 조건 1: 부재 부호 화이트리스트에 있는가?
        if content in ["VG1", "WG1", "MT1", ...]:
            # 조건 2: 사용자가 친 박스 안에 있는가?
            if 287652.6 <= x <= 329111.3 and 1182320.7 <= y <= 1214368.0:
                counter[content] += 1`

이 두 조건을 통과한 텍스트만 Counter로 집계해요. 결과:

`VG1: 19개
MT1: 15개
VT1: 13개`

### 비유로 정리

이걸 비유로 설명하면 **지도와 실제 지역**의 관계예요:

1. **실제 지역**(= DXF 좌표계)에는 모든 건물이 진짜 GPS 좌표를 갖고 있어요
2. **지도**(= PNG)는 그 지역을 줄여서 종이에 그린 것
3. 사용자가 지도 위에 손가락으로 동그라미 치면(= 픽셀 박스), 우리는 그게 **실제 어느 지역**을 가리키는지 비례식으로 계산할 수 있어요
4. 그 다음 진짜 GPS 데이터베이스(= DXF 파일)에서 그 영역 안의 건물(= 부재 부호)만 골라 세는 거예요

**지도와 실제 좌표 사이의 비례 관계만 정확히 알고 있으면**, 사람이 직관적으로 그린 박스도 정밀한 데이터 추출로 연결될 수 있어요. 그게 이번 PoC의 핵심이에요.
